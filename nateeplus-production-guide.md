# คู่มือการนำระบบ NaTee Plus ไปใช้งานจริงบน nateeplus.com และการตั้งค่าหน้า Share อย่างละเอียด
*(NaTee Plus Production Deployment & Share Settings Guide)*

คู่มือนี้จัดทำขึ้นอย่างละเอียดและอธิบายเป็นขั้นตอนอย่างช้า ๆ เพื่อช่วยลดความกังวลและทำให้ท่านสามารถนำระบบที่พัฒนาสำเร็จจากระบบทดสอบ (Sandbox) ไปเปิดใช้งานบนเซิร์ฟเวอร์จริงภายใต้โดเมน **nateeplus.com** ได้อย่างปลอดภัย 100% โดยข้อมูลไม่สูญหายค่ะ

---

## ส่วนที่ 1: ขั้นตอนการตั้งค่าในหน้า "Share" ของ AI Studio (ต้องกดตรงไหน?)

เมื่อท่านต้องการแชร์แอปพลิเคชันเวอร์ชันทดสอบนี้ให้ผู้อื่น หรือต้องการตั้งค่าเพื่อเปิดระบบอย่างปลอดภัย ให้คลิกที่ปุ่ม **"Share"** ที่มุมขวาบนของหน้าจอ AI Studio แล้วตั้งค่าตามขั้นตอนดังนี้ค่ะ:

### 1. การควบคุมการเข้าถึงระบบ (General Access)
*   **มองหาหัวข้อ:** "General access" (การเข้าถึงทั่วไป) ซึ่งจะมีสัญลักษณ์รูปแม่กุญแจ 🔒
*   **สิ่งที่ต้องกด:** ให้คลิกที่เมนูตัวเลือก (Dropdown) ด้านล่างคำว่า General access:
    *   **หากต้องการเปิดเป็นสาธารณะ (Public):** ให้เลือกเปลี่ยนจาก **"Restricted"** (จำกัดเฉพาะอีเมล) เป็น **"Anyone with the link"** (ทุกคนที่มีลิงก์สามารถเข้าใช้งานได้) สมาชิกและแอดมินคนอื่น ๆ จึงจะสามารถเปิดลิงก์และใช้งานระบบทดสอบนี้ได้ค่ะ
    *   **หากต้องการจำกัดเฉพาะทีมงาน:** ให้เลือก **"Restricted"** แล้วพิมพ์อีเมลของทีมงาน (เช่น พิมพ์เพิ่มในช่อง "Add people and groups") เพื่อระบุรายบุคคลที่ต้องการให้สิทธิ์เปิดแอปพลิเคชันนี้

### 2. การอนุญาตให้คนอื่นลงชื่อเข้าใช้งาน (People who can sign in)
*   **มองหาหัวข้อ:** "People who can sign in to your app" (บุคคลที่สามารถลงชื่อเข้าใช้งานแอปได้)
*   **สิ่งที่ต้องทำ:** 
    1.  พิมพ์อีเมลของแอดมินหรือสมาชิกทดสอบที่ท่านต้องการให้อนุญาตให้สิทธิ์ลงชื่อเข้าใช้ (สามารถแชร์ได้สูงสุด 99 อีเมลสำหรับช่วงทดสอบ)
    2.  กดปุ่ม **"Add"** หรือ **"Invite"** เพื่อให้ระบบลงทะเบียนอีเมลเหล่านั้นในระบบความปลอดภัย

### 3. การแสดงผลแบบเต็มหน้าจอ (Default to Fullscreen)
*   **สิ่งที่ต้องกด:** ในหัวข้อ Settings ด้านล่าง ให้เปิดสวิตช์ (Toggle) ตัวเลือก **"Default to fullscreen"** เป็นสีเขียว (เปิดใช้งาน) เพื่อให้ทุกคนที่กดเปิดลิงก์แอปพลิเคชันของท่าน จะแสดงผลแบบเต็มหน้าจอเว็บเบราว์เซอร์ทันทีอย่างสวยงาม โดยไม่มีหน้าต่างแชตของ AI รบกวนสายตาค่ะ

## ส่วนที่ 2: วิธีการอัปเดตระบบจริงให้เหมือนกับตัวทดสอบ (ความกังวลสูงสุดของท่าน)

การทำให้ระบบจริงบนโดเมน **nateeplus.com** ทำงานได้เหมือนกับระบบทดสอบที่พัฒนาเสร็จแล้วอย่างสมบูรณ์แบบ มีขั้นตอนการทำดังนี้ค่ะ:

### ขั้นตอนที่ 1: การดาวน์โหลดซอร์สโค้ด (Source Code Export)
1.  ที่แถบเมนูด้านซ้ายหรือขวาใน AI Studio (ปุ่มฟันเฟืองหรือเมนูตั้งค่า Project Settings)
2.  เลือกคำสั่ง **"Export to GitHub"** (เพื่อเชื่อมและอัปโหลดโค้ดขึ้น GitHub ของท่านโดยตรง) หรือเลือก **"Download ZIP"** เพื่อบันทึกโค้ดทั้งหมดลงในเครื่องคอมพิวเตอร์ของท่าน
3.  โค้ดชุดนี้คือโค้ดเวอร์ชันล่าสุดที่ถูกปรับปรุงระบบความปลอดภัยเรียบร้อยแล้ว (ไม่มีการเขียน API Key หรือค่าคอนฟิกของระบบทดสอบลงในโค้ดตรง ๆ เพื่อป้องกันการสับสน)

### ขั้นตอนที่ 2: การทำงานของฐานข้อมูลแยกส่วน (Production vs Sandbox Database)
ระบบที่เราเขียนไว้รองรับโครงสร้างแบบ **Multi-Database (แยกแง่การทำงาน)** โดยใช้ **Cloud Firestore** ชุดเดียวกันที่สิงคโปร์ (`asia-southeast1`) แต่แยกกล่องข้อมูลออกเป็น 2 ส่วนเด็ดขาดผ่านการตั้งค่าในระบบหลังบ้าน:
*   **ระบบจริง (Production):** ข้อมูลจะถูกเก็บไว้ใน Collection ชื่อ `app_sections`
*   **ระบบจำลอง (Sandbox):** ข้อมูลจะถูกเก็บไว้ใน Collection ชื่อ `app_sections_sandbox`
*   **การอัปเดตระบบจำลอง:** ท่านสามารถกดปุ่ม **"Reset from Production"** บน Admin Dashboard ในโหมด Sandbox ระบบจะทำการสำเนา (Clone) ข้อมูลสมาชิก สินค้า และยอดเงินล่าสุดจากระบบจริงลงไปในกล่องจำลองโดยอัตโนมัติ เพื่อให้ท่านทดสอบระบบคำนวณคะแนนและแผนธุรกิจได้อย่างปลอดภัย 100% โดยไม่มีผลกระทบต่อเงินจริงของระบบหลักค่ะ

### ขั้นตอนที่ 3: การ Deploy บน Firebase App Hosting
เมื่อท่านนำโค้ดที่ดาวน์โหลดหรืออัปขึ้น GitHub ไปเชื่อมต่อกับ **Firebase App Hosting** ของโดเมน `nateeplus.com` ให้เข้าไปตั้งค่าตัวแปรสภาพแวดล้อม (Environment Variables) ในหน้า Firebase Console ของโครงการจริงดังนี้เพื่อความปลอดภัย:

ให้ระบุค่าคอนฟิกของระบบจริงลงใน Environment Variables ของ Firebase App Hosting โดยไม่ต้องเขียนลงในไฟล์โค้ด:
1.  `FIREBASE_API_KEY` (API Key ของระบบจริง)
2.  `FIREBASE_AUTH_DOMAIN` (`nateeplus.com` หรือ Domain จริง)
3.  `FIREBASE_PROJECT_ID` (รหัสโปรเจกต์จริงบน Firebase)
4.  `FIREBASE_STORAGE_BUCKET` (Storage Bucket ของระบบจริง)
5.  `FIREBASE_MESSAGING_SENDER_ID` (Messaging Sender ID ของระบบจริง)
6.  `FIREBASE_APP_ID` (App ID ของระบบจริง)
7.  `FIRESTORE_DATABASE_ID` (หากต้องการระบุฐานข้อมูลย่อยเพิ่มเติม เช่น `(default)` หรือชื่อฐานข้อมูลที่สร้างขึ้น)

> **💡 ข้อดีของวิธีนี้:** โค้ดโปรเจกต์เดียวกันนี้จะทำงานเป็น **ระบบจริง** ทันทีเมื่อดึงการตั้งค่าจาก Environment Variables ของระบบจริงใน Firebase Console และจะทำงานเป็น **ระบบทดสอบ** เมื่อรันอยู่ใน AI Studio โดยโครงสร้างหน้าตาและการคำนวณทั้งหมดจะถอดแบบและทำงานเหมือนกัน 100% ค่ะ

## ส่วนที่ 3: วิธีการซิงค์และย้ายข้อมูลจากระบบทดสอบไปสู่ระบบจริง
เมื่อท่านทดสอบระบบจนพอใจแล้วและต้องการให้สมาชิกลงทะเบียนจริง:
1.  **การล้างระบบเพื่อเริ่มใช้งานจริง:** เข้าสู่เมนูผู้ดูแลระบบ (Admin Dashboard) เลือกปุ่ม **"System Reset (ล้างระบบเริ่มต้นใหม่)"** เพื่อล้างข้อมูลสมาชิกและประวัติธุรกรรมจำลองทั้งหมดออกไป เพื่อเตรียมรับข้อมูลสมาชิกลงทะเบียนและยอดเงินโอนจริง
2.  **ความปลอดภัยด้านธุรกรรม:** ทุกยอด E-Cash, E-Money, และ E-Coupon ที่แอดมินหรือผู้ใช้สั่งปรับเปลี่ยนบนระบบหลังบ้าน จะถูกบันทึกประวัติ (Transaction Logs) ลงในสมุดบัญชีแยกประเภทอย่างละเอียด ทำให้ท่านตรวจสอบย้อนหลังที่มาของตัวเลขทุกเศษสตางค์ได้เสมอค่ะ
